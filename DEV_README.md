# Developer README

This file explains how the repository works from the maintainer/developer side.
The public `README.md` is edited manually and is not generated by automation.

## Repository layout

- `README.md` - public project description, edited manually.
- `DEV_README.md` - developer notes for maintainers.
- `data/students/<github-username>.txt` - source files submitted by students.
- `students.md` - generated contribution table. Do not edit manually.
- `scripts/validate_student_pr.py` - validates student pull requests.
- `scripts/build_markdown.py` - builds generated markdown files.
- `tests/` - unit tests for the scripts.
- `.github/workflows/validate_student_pr.yml` - PR validation workflow.
- `.github/workflows/build_markdown.yml` - generated markdown build workflow.
- `.github/workflows/tests.yml` - regular test workflow.

## Data model

Each student gets one file:

```text
data/students/<github-username>.txt
```

The filename must match the GitHub username of the PR author. Each non-empty
line must be one GitHub pull request or issue URL:

```text
https://github.com/OWNER/REPO/pull/123
https://github.com/OWNER/REPO/issues/123
```

`students.md` is generated from these files. The builder calls the GitHub API
for every link and refreshes the current contribution metadata:

- contributor profile URL
- project URL
- type: `PR` or `Issue`
- status: `Open`, `Closed`, or `Merged`
- topic title and URL

Topic titles are cleaned before rendering. For example, a leading CPython-style
prefix such as `gh-69619:` is removed.

## Workflow split

### `validate_student_pr.yml`

`validate_student_pr.yml` validates untrusted student pull requests before merge.

It runs on:

```yaml
pull_request_target:
  types:
    - opened
    - synchronize
    - reopened
    - edited
```

It uses read-only permissions:

```yaml
permissions:
  contents: read
  pull-requests: read
```

It checks out the trusted base branch version of the validator:

```yaml
ref: ${{ github.event.pull_request.base.sha }}
persist-credentials: false
```

This is important because `pull_request_target` runs with permissions from the
base repository. The workflow must not execute code from the contributor fork.

The validator enforces:

- the PR changes exactly one student file
- the file path is `data/students/<github-username>.txt`
- the filename username matches the PR author
- all links are GitHub PR or issue links
- each linked PR or issue was opened by the same GitHub user
- duplicate links are rejected

This workflow does not generate or commit files.

When validation fails, the script exits with status `1` and prints a GitHub
Actions error annotation with:

- the exact reason the PR failed
- a `How to fix:` instruction for the student

GitHub Actions cannot prevent a pull request from being opened. It can fail the
PR check. To prevent invalid PRs from being merged, the repository must require
the `Validate student PR / Validate student PR` status check on the `main`
branch.

### `build_markdown.yml`

`build_markdown.yml` runs only after trusted changes are on `main`, or when a
maintainer runs it manually.

It runs on:

- push to `main` changing `data/students/**`
- push to `main` changing `data/my.md`
- push to `main` changing the builder script or workflow
- manual `workflow_dispatch`

It has write permission:

```yaml
permissions:
  contents: write
```

It runs:

```bash
python3 scripts/build_markdown.py
```

Then it commits `students.md` only if the generated output changed.

This workflow is responsible for refreshing statuses. If a PR was open before
and later becomes merged, running this workflow again updates `students.md`.

### `tests.yml`

`tests.yml` runs the Python tests on pull requests and pushes to `main`.

It does not commit anything. It only verifies the code.

## Local development

Run tests:

```bash
.venv/bin/python -m pytest -q
```

Generate `students.md` locally using the GitHub CLI token:

```bash
GITHUB_TOKEN="$(gh auth token)" python3 scripts/build_markdown.py
```

Check formatting problems in diffs:

```bash
git diff --check
```

## Expected contribution flow

1. Student forks the repository.
2. Student adds or edits `data/students/<their-github-username>.txt`.
3. Student opens a PR to `main`.
4. `validate_student_pr.yml` validates the PR.
5. Maintainer merges the PR.
6. The merge creates a push to `main`.
7. `build_markdown.yml` regenerates `students.md` and commits it if needed.

## Maintainer notes

- Do not make `README.md` generated unless the project policy changes.
- Do not edit `students.md` manually; edit `data/students/*.txt` instead.
- Keep validation and generation separate:
  - validation protects the repo from invalid PR data
  - generation refreshes derived markdown after trusted changes
- Be careful with `pull_request_target`; it should only run trusted code from
  the base branch.
- Keep branch protection enabled for `main` and require the validator check, or
  GitHub will allow maintainers to merge invalid PRs despite the failed check.
